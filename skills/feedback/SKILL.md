# Feedback

Turn project experience into durable harness improvements.

## Inputs

Read when available:

- control/feedback-log.md
- retrospectives
- failed gate reports
- review comments
- defects
- incidents
- repeated corrections

## Output

Create or update:

- control/feedback-log.md
- docs/10-retro/harness-improvements.md

## Classify Feedback

- missing context
- missing template
- missing gate
- missing skill
- weak adapter
- weak naming
- weak traceability
- weak approval policy
- stale documentation
- repeated agent error

## Improvement Recommendation

For each feedback item, recommend one:

- update docs
- update template
- add gate
- update skill
- add adapter mapping
- add test
- add checklist
- add human approval rule

## Rule

If the same issue appears twice, do not leave it as advice. Convert it into a mechanical rule, template, gate, or skill update.
